# Claude zh-CN Manager v1 规格

日期：2026-07-03

## 1. 背景

Claude Desktop 的界面文本来自多个来源：本地资源文件、Electron 桌面壳、远程 `claude.ai` 页面，以及账号状态相关的动态文本。单独替换一个语言文件无法覆盖完整体验。

Claude zh-CN Manager 的定位是把这些来源收敛到一套稳定流程里：用户通过一个项目入口完成部署、启动、登录回调桥接、更新重建、翻译增量维护、诊断和回滚。

## 2. 目标

v1 的目标是做一个稳定、可维护、可回滚的 Claude 中文化管理器。

核心能力：

1. 一键启动 Claude zh-CN。
2. 一键检查官方版本。
3. 一键更新官方 Claude 并重新应用汉化。
4. 保留基础翻译资源和补丁逻辑，作为基础汉化层。
5. 支持本地增量翻译，用来覆盖基础资源没有翻译或官方新增的内容。
6. 采集仍然显示英文的可见文本，生成待翻译清单。
7. 重新生成远程页面 DOM 翻译表，并注入到 Claude Desktop。
8. 安装并验证 OAuth 登录回调桥接器。
9. 每次危险操作前备份，失败时保留上一个可用版本。

不追求的目标：

- 不承诺官方远程页面新增文本出现的同一秒就自动完美翻译。
- 不修改 Anthropic 账号、服务端数据、订阅或权限。
- 不直接依赖官方语言菜单出现中文选项，因为当前主界面主要来自远程 `claude.ai`，语言菜单不是唯一控制点。

## 3. 为什么仍需要启动器

启动器不是单纯“打开软件”的快捷方式，它是稳定控制点。

这是什么：启动器负责指定汉化版 Claude、独立用户数据目录、登录回调、语言配置和补丁检查。

为什么要这样做：官方更新会覆盖 `Claude.exe`、`app.asar`、本地资源、preload 注入和快捷方式。没有启动器，更新后需要用户重新确认多个入口。

会影响哪里：桌面快捷方式、开始菜单入口、`claude://` 登录回调、用户数据目录、汉化补丁执行顺序。

不做会怎样：用户需要手动排查路径、注册协议、运行脚本，整体部署成本会升高。

v1 继续使用 PowerShell 管理器作为启动器入口。后续可以升级为图形界面，但不作为 v1 必需项。

## 4. 总体架构

v1 分为五层：

1. 入口层：`claude-zh-manager.ps1`
   - 提供菜单。
   - 解释每个操作会影响什么。
   - 统一调用下面的功能脚本。

2. 路径配置层：`config/paths.local.json`
   - 保存本机路径，例如补丁工具、汉化版 Claude、用户数据目录、启动器路径。
   - 迁移到新的项目目录时，只需要更新这里。

3. 基础汉化层：补丁工具与基础资源
   - 复用基础翻译资源、硬编码补丁、启动器创建和便携版处理能力。
   - 基础资源是底座，本地增量和远程页面翻译是补充层。

4. 本地增量层：`overrides/`
   - 保存用户确认过的新增翻译。
   - 任何官方新增功能、设置项、错误提示，都优先进入这里。
   - 更新官方版本时，本地增量层最后应用，避免被基础资源覆盖。

5. 远程页面运行时层：DOM 翻译注入
   - 处理登录后来自 `https://claude.ai/` 的主页面文本。
   - 通过可见文本匹配，把英文替换成中文。
   - 对动态出现的设置页、弹窗、错误提示持续监听。

## 5. 一键更新并重新汉化流程

用户操作路径：

1. 打开 `Claude zh-CN Manager`。
2. 选择“检查官方更新”。
3. 如有新版本，选择“一键更新并重新汉化”。
4. 管理器完成下载、重建、补丁、验证。
5. 用户从同一个入口启动 Claude zh-CN。

内部流程：

1. 关闭正在运行的 Claude 进程。
2. 备份当前可用汉化版：
   - `Claude.exe`
   - `resources/app.asar`
   - 关键资源 JSON
   - `claude://` 注册表回调
3. 调用补丁工具更新能力，下载并重建新版本便携 Claude。
4. 应用基础汉化资源。
5. 应用本地增量翻译。
6. 强制 `en-US` 资源入口加载中文资源，用来处理 Claude 把语言改回英文的情况。
7. 重新注入远程 `claude.ai` DOM 翻译补丁。
8. 重新安装 OAuth 回调桥接器。
9. 清理必要缓存，但不删除账号登录数据。
10. 输出诊断报告：
    - 当前 Claude 版本
    - 汉化版路径
    - 登录回调状态
    - 本地资源覆盖率
    - 远程 DOM 翻译表条数
    - 待翻译文本数量

失败处理：

- 下载失败：保留旧版本，不替换当前汉化版。
- 补丁工具失败：保留旧版本，并提示失败步骤。
- DOM 注入失败：保留本地资源汉化结果，但提示主页面可能仍有英文。
- 回调写入失败：不删除旧回调备份，提示需要管理员权限或手动配置。

## 6. 新增英文文本维护方案

要持续维护完整中文体验，不能只靠一次性搜索替换。原因是 Claude 的文字来源不止一个地方。

文本来源分四类：

1. 本地资源 JSON：菜单、部分配置页、桌面壳文案。
2. `app.asar` 硬编码文本：少量写死在程序里的菜单或提示。
3. 远程 `claude.ai` 页面：登录后的主页、设置页、弹窗、错误提示。
4. 动态服务端文本：模型、计划、权限、错误状态等随账号或网络状态变化的内容。

v1 使用“采集、确认、覆盖、验证”的闭环：

1. 采集
   - DOM 翻译补丁记录未命中的可见英文文本。
   - 本地资源扫描器比较 `en-US` 与 `zh-CN`，输出未翻译 key。
   - 日志扫描器读取 Claude 日志中的英文错误提示。

2. 确认
   - 生成 `overrides/pending-*.json` 待翻译文件。
   - 人工确认中文翻译，避免把产品名、快捷键、代码片段误翻译。

3. 覆盖
   - 确认后的内容进入：
     - `overrides/frontend-zh-CN.override.json`
     - `overrides/desktop-zh-CN.override.json`
     - `overrides/statsig-zh-CN.override.json`
     - `overrides/remote-dom-zh-CN.override.json`

4. 验证
   - 重新生成远程 DOM 翻译表。
   - 重新注入补丁。
   - 输出剩余英文清单。

这套机制的结论是：已知文本可以做到稳定覆盖；官方新增文本会先进入待翻译清单，确认后纳入增量包。这样即使基础资源暂时不更新，本地项目也能继续补新功能文案。

## 7. 数据文件设计

新增或规范以下文件：

- `overrides/remote-dom-zh-CN.override.json`
  - 用于登录后远程页面的精确文案翻译。

- `overrides/pending-remote-dom.json`
  - 运行时采集到但还没有确认翻译的远程页面英文文本。

- `overrides/pending-local-resources.json`
  - 本地资源里还没有中文覆盖的 key。

- `reports/latest-diagnose.json`
  - 最近一次诊断结果。

- `reports/latest-untranslated.json`
  - 最近一次待翻译文本清单。

- `backups/update-YYYYMMDD-HHMMSS/`
  - 每次更新前的可回滚备份。

## 8. 菜单设计

v1 菜单建议保留当前 PowerShell 形式，但调整为更清晰的任务分组。

日常使用：

1. 启动 Claude zh-CN
2. 检查状态
3. 检查官方更新
4. 一键更新并重新汉化

维护工具：

5. 配置 `claude://` 回调
6. 安装 OAuth 回调桥接器
7. 强制英文入口加载中文资源
8. 重新注入远程页面汉化

翻译维护：

9. 扫描待翻译文本
10. 应用本地增量翻译
11. 生成汉化覆盖报告

备份回滚：

12. 查看最近备份
13. 回滚到上一个可用汉化版

## 9. 角色影响

产品视角：

- 目标是把“脚本集合”变成“可重复使用的工具”。
- v1 优先保证更新、汉化、回滚闭环，不做复杂图形界面。

设计视角：

- 用户入口从多个脚本收敛到一个管理器。
- 每个菜单项需要说明影响范围，降低误操作成本。
- 后续如果做图形界面，信息结构可以沿用 v1 菜单分组。

前端/脚本视角：

- 需要把现有脚本能力编排成稳定流程。
- 需要新增翻译采集、报告生成、远程 DOM override 合并。

测试视角：

- 重点验证更新后是否仍能启动、登录、显示中文。
- 重点覆盖设置页、错误弹窗、网络异常、官方新增版本。

## 10. 验收标准

v1 完成后应满足：

1. 从管理器启动 Claude zh-CN，登录后主界面保持中文。
2. `claude://` 回调指向当前项目配置的桥接器或启动器。
3. 一键更新流程不会删除旧可用版本。
4. 基础翻译、本地增量翻译、远程 DOM 翻译按固定顺序应用。
5. 官方新增文本可以被采集到待翻译文件。
6. 已确认的新增翻译在下一次启动或重新注入后生效。
7. 诊断报告能说明当前失败点，而不是只显示“失败”。
8. 存在明确回滚入口，可以恢复到最近一次可用汉化版。

## 11. 风险与边界

主要风险：

- 官方更新改变 Electron 或资源结构，导致旧补丁位置失效。
- 远程 `claude.ai` 页面改版，导致 DOM 翻译匹配率下降。
- 账号状态、地区、订阅状态不同，看到的英文文本不同。
- 注册表写入可能受权限限制。

对应处理：

- 所有补丁前先备份。
- 注入脚本必须带 marker，避免重复注入。
- 待翻译采集不直接自动覆盖，先进入 pending 文件。
- 诊断报告明确区分本地资源失败、远程页面失败、登录回调失败。

## 12. 后续实施顺序

建议分四个稳定阶段提交：

1. 整理管理器菜单和诊断报告。
2. 打通一键更新并重新汉化流程。
3. 增加待翻译文本采集和 override 合并。
4. 增加备份查看与一键回滚。

每个阶段都应独立可运行、可提交、可回滚。
